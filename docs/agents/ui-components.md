# UI Components & Styling

Stack: **Tailwind CSS v4 + shadcn/ui** (`new-york` / `neutral`, per `components.json`). No CSS-in-JS. The design system is a single token file: `globals.css`.

## THE rule: style only from semantic theme tokens

Every color, size, and radius comes from a CSS variable in `globals.css`, mapped to a Tailwind utility via `@theme inline` (e.g. `--color-primary` ‚Üí `bg-primary`). Re-skinning the app means editing that one file. Therefore, in app code:

- Use semantic token utilities: `bg-background`, `text-foreground`, `text-muted-foreground`, `border-border`, `bg-primary`, `bg-destructive`, and the project-added `bg-success` / `bg-warning`.
- **Never hard-code a color/size/radius.** Two forms are both banned because both bypass the central theme:
    - bracketed arbitrary values ‚Äî `bg-[#f00]`, `text-[13px]`, `rounded-[7px]`
    - **raw Tailwind palette classes ‚Äî `text-green-700`, `bg-red-500`** (easy to miss: these are NOT caught by lint, see below)
- If a token is missing, add it to `globals.css` rather than reaching for a raw value.

```tsx
// ‚úÖ token            // ‚ùå arbitrary        // ‚ùå raw palette (lint won't catch)
text-muted-foreground   text-[#71717a]         text-zinc-500
bg-success              bg-[#16a34a]           bg-green-600
```

### Lint guard ‚Äî and its blind spot

`eslint.config.js` has a `no-restricted-syntax` rule that **fails the build** on bracketed arbitrary values anywhere under `src/web/**`, exempting `src/web/components/ui/**`. **It only matches bracketed `[...]` syntax ‚Äî it does NOT flag raw palette classes like `text-green-700`.** You must self-enforce the no-raw-palette half of the rule; CI won't.

## Adding design tokens

Add to `globals.css` only, in all three places: the variable under `:root`, its dark value under `.dark`, and the mapping in `@theme inline`. (This is how `success`/`warning` were added.)

## Helpers

- `cn()` for conditional/composed `className`s (static strings don't need it).
- Fixed variant sets (sizes/intents): `cva` + `VariantProps`. The vendored `Button` primitive is the canonical pattern ‚Äî don't hand-roll `if/else` className props instead.
- Class sorting is automatic via `prettier-plugin-tailwindcss`; don't hand-sort.

## Vendored `components/ui/` is third-party

Generated by the shadcn CLI. Treat as vendored:

- Don't hand-edit them to restyle ‚Äî change look-and-feel via `globals.css` tokens, or wrap/compose them.
- Exempt from the arbitrary-value lint rule (their `ring-[3px]`, `[&_svg]:size-4` are upstream).
- Add new primitives by re-running the CLI, not by authoring by hand.

## Dark mode

Dark tokens exist under `.dark`, but **no toggle ships** (enable by adding `.dark` to `<html>`). Use semantic tokens so dark mode works for free; avoid `bg-white` / `text-black`.

## No inline `style={{}}`

Use utilities/tokens, never inline styles.

## Custom CSS (escape hatch)

Default is utility classes ‚Äî no stylesheet per component, so deleting a component deletes its styles. When a utility genuinely can't express something:

- **Co-locate a CSS Module** ‚Äî `Foo.module.css` next to `Foo.tsx`, imported into it. Scoping prevents leaks and the file is deleted with the component, so custom class names never orphan. Drive it from theme tokens via the `:root` vars ‚Äî `var(--primary)`, `var(--success)`, etc. (not `var(--color-*)`, which `@theme inline` doesn't emit); never hard-code colors.
- **Don't add custom classes to `globals.css`** ‚Äî it's for design-system tokens only (`@theme`, `:root`/`.dark`, the base layer). Loose global classes accumulate and orphan; that's what co-location avoids. A pattern shared across components becomes a shared component + `cva`, not a global class.

(No plain global `*.css` per component ‚Äî it leaks. No CSS-in-JS.)
